# 新闻源优化记录

记录时间：2026-06-08

## 当前结论

这次中天科技案例说明：新闻源本身不一定慢，但本地项目存在抓取频率、补抓和硬催化识别不足的问题。

- 财联社原始快讯：2026-06-05 16:10 左右已发布中天科技中标消息。
- 本地 `data_dev/cls/2026-06-05.json`：文件写入时间约为 2026-06-05 14:46，只保存到 14:45 左右的快讯。
- 本地 `data_dev/cls/2026-06-08.json`：2026-06-08 08:40 重新拉取后，才在“财联社6月8日早间新闻精选”汇总里看到中天科技。

准确问题：6月5日收盘后没有继续抓取，也没有在6月8日补抓6月5日16:10的原始快讯。

## 本次已补充

新闻保存前统一补充本地抓取/入库时间字段：

- `crawled_at`：本地抓取/入库时间，ISO 格式，带北京时间时区。
- `crawled_at_display`：本地抓取/入库时间，方便阅读。
- `crawled_date_bj`：本地抓取日期。
- `crawled_at_source`：抓取时间来源。
- `latency_minutes`：本地抓取时间 - 新闻发布时间，单位分钟。
- `latency_level`：延迟等级，分为 `fast`、`normal`、`slow`、`stale`、`clock_skew`、`unknown`。
- `latency_note`：延迟计算说明。

历史 JSON 没有真实单条抓取时间，只能用文件原修改时间近似回填，并标记：

- `crawled_at_source = file_mtime_backfill`

## 周六优化清单

### 0. 周末部署优先级

第一版目标不是继续扩很多新闻源，而是先做成“服务器能稳定自动跑、接口能稳定给股票分析系统调用、新闻只保留一周”的可用版本。

本阶段先保留现有 4 个源：

- `cls`
- `eastmoney_global`
- `cninfo`
- `ndrc`

暂时不急着新增证券时报和每经。等服务器稳定跑一周后，再根据 `latency_minutes` 和漏抓案例决定是否扩源。

必须先完成：

- 一键启动入口稳定。
- 定时采集和接口读取同一个数据目录。
- 服务器部署命令清晰。
- 健康检查能看出各源是否断流。
- 启动时能补抓最近几天。
- 失败不影响整个服务运行。

建议完成：

- 长新闻拆条。
- 公司级硬催化识别。
- 接口返回 source summary 和 newest time。
- 部署文档。
- systemd/supervisor 守护进程配置。

### 1. 抓取频率

交易日建议：

- 09:00-11:30：每 3-5 分钟抓一次快讯。
- 13:00-15:30：每 3-5 分钟抓一次快讯。
- 15:30、16:30、17:30、20:30、22:30：重点抓公告、晚间快讯、公司新闻。
- 07:30、08:30、09:05：盘前强制刷新。

周末建议：

- 每天 09:00、14:00、20:30 抓一次。
- 周日 20:30 和周一 07:30 必须刷新并做补抓。

### 2. 补抓机制

每次启动项目时，先补抓最近 3 天新闻，不只抓当天。

重点解决：

- 周五收盘后消息。
- 周末政策和产业事件。
- 周一早盘汇总之前的原始快讯。

### 3. 新增新闻源

建议追加：

- `stcn`：证券时报/人民财讯，补强 A 股公告、信披、公司硬催化。
- `nbd`：每日经济新闻/每经AI快讯，补强广覆盖快讯、公司新闻、宏观和产业事件。

保留现有：

- `cls`：财联社，短线快讯和题材雷达。
- `eastmoney_global`：东方财富全球财经快讯，海外、商品、汇率、宏观。
- `cninfo`：巨潮公告，权威公告源。
- `ndrc`：发改委政策源。

### 4. 公司级硬催化提取

必须单独识别并提高优先级：

- 中标
- 合同
- 订单
- 采购
- 量产
- 获批
- 涨价
- 回购
- 增持
- 减持
- 监管
- 澄清
- 风险提示

像“中天科技中标15.18亿元数据中心用MPO光纤跳线项目”这种消息，必须独立进入候选池，不能只埋在“早间新闻精选”长文本里。

### 5. 长新闻拆条

对以下类型新闻做拆条：

- 财联社早间新闻精选
- 晚间公告精选
- 盘前要闻
- 公告精选
- 证券时报/每经汇总快讯

拆条后每条单独打分、映射股票、识别风险。

### 6. 去重和交叉验证

同一新闻多源出现时：

- 保留最早 `publish_time_bj`。
- 记录所有确认来源。
- 保留最完整的原文链接。
- 不重复写入接口结果。

### 7. 接口输出建议

给股票分析系统的新闻接口建议返回：

- 新闻标题
- 新闻发布时间
- 本地抓取时间
- 延迟分钟数
- 来源
- 原文链接
- 摘要/正文片段
- 股票代码和股票名称
- 主题标签
- 风险标签
- 是否公司级硬催化
- 影响力评分

不要大量对外分发全文，优先提供标题、摘要、链接、结构化字段，降低版权风险。

## 周末服务器部署 MVP 清单

### 1. 启动入口统一

`run.py` 应该作为服务器唯一启动入口，同时启动：

- 新闻 HTTP 接口。
- 后台定时采集任务。

重点检查：接口读取目录和爬虫写入目录必须一致，避免接口读 `data_dev`、采集写 `data` 这种错位。

推荐服务器启动形式：

```bash
python run.py --host 0.0.0.0 --port 8765 --data-dir data --retention-days 7
```

本地开发可以继续用：

```bash
python run.py --host 127.0.0.1 --port 8765 --data-dir data_dev --retention-days 7
```

### 2. 数据目录和环境配置

服务器正式数据目录建议使用 `data`。

本地开发数据目录建议使用 `data_dev`。

如果 `config.py` 的 development/production 配置会自动覆盖目录，需要确认：

- `DATA_COLLECTOR_ENV=production` 时默认写 `data`。
- 命令行 `--data-dir` 优先级最高。
- 接口、爬虫、去重索引、增量状态都跟随同一个数据目录。

### 3. 启动补抓

服务器启动后先补抓最近 3 天，而不是只抓当天。

原因：

- 覆盖周五收盘后公告。
- 覆盖周末政策和产业事件。
- 覆盖服务器停机期间漏掉的快讯。
- 周一早盘前能拿到更完整的消息面。

建议默认：

```text
启动立即抓取：最近 3 天
定时循环抓取：最近 1 天或按各源增量状态抓取
```

### 4. 健康检查增强

`/health` 不应只返回 `ok`，后续建议返回每个源状态：

- `source`
- `latest_publish_time`
- `latest_crawled_at`
- `latest_latency_minutes`
- `items_24h`
- `last_error`
- `status`: `ok` / `stale` / `error`

这样下游股票分析系统可以判断新闻源是否可靠。

### 5. 新闻接口增强

`/news` 建议保留当前简单用法，同时增强响应元信息：

- `count`
- `source`
- `keyword`
- `retention_days`
- `newest_time`
- `newest_crawled_at`
- `sources_summary`
- `data`

`sources_summary` 示例：

```json
{
  "cls": {"count": 120, "newest_time": "...", "latest_latency_minutes": 3.2},
  "eastmoney_global": {"count": 200, "newest_time": "...", "latest_latency_minutes": 8.5}
}
```

### 6. 失败不中断

某一个源失败时，整个服务不能挂。

需要做到：

- 记录失败源。
- 记录错误原因。
- 继续运行其他源。
- 下一个周期自动重试。
- 连续失败时在 `/health` 里显示 `error` 或 `stale`。

### 7. 日志和排障

部署后至少要能快速回答：

- 服务是否活着。
- 最近一次抓取是什么时候。
- 每个源最新新闻是什么时候。
- 哪个源失败了。
- 错误原因是什么。
- 接口是否正常返回。

建议补充 `DEPLOY.md`，包含：

- 安装依赖。
- 启动命令。
- 停止命令。
- 查看日志。
- 健康检查。
- 接口调用示例。
- 常见问题。

### 8. 守护进程

周末服务器部署建议用 `systemd` 或 `supervisor` 托管：

- 服务器重启后自动启动。
- 进程异常退出后自动拉起。
- 日志统一写入文件。

第一版可以先手动启动验证，稳定后再加守护配置。
